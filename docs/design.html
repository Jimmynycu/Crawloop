1Goals & constraints

Goal. Crawl and extract correct structured data from many differently-built websites. Fast path: a registry of per-page-family crawlers (one crawler per page template). When a crawler breaks or returns wrong/empty data, The Loop automatically generates a new versioned crawler for that same page family — it never overwrites or deletes the old one (see §7) — proves it correct against real pages, and registers it as the family's new active version. The next request uses it immediately; old versions are kept for rollback and as a runtime fallback ladder.

"Generate a replacement" = append a new version, not a new family, not a delete. A page family (e.g. books.toscrape.com/product_list) is a stable identity. Inside it lives an ordered list of crawler versions v1, v2, v3…, each an immutable code artifact. The Loop adds v(N+1) and points the family's active_version at it. The old code stays on disk for one-command rollback and as a fallback rung (§7).

Hard constraints (enforced in code, not just docs):

  • Hard domain allowlist = the ethical backbone. The engine can only ever fetch domains explicitly listed in authorized_domains.yaml — nothing else, ever. This is the structural guarantee that everything below stays inside your authorized scope.
  • Access blocks are recoverable inside that scope. v2 429 / 401 / 403 / Cloudflare / captcha are no longer a dead stop. They become a recoverable failure class the system resolves by legitimate means — backoff, your credentials/sessions, owner-issued bypass tokens, browser emulation (§6 Access layer). "Solve the block" is in-scope precisely because the allowlist guarantees the target is one you're authorized on.
  • robots.txt — v2 config toggle respect_robots, default OFF for the POC (your call; targets are owned/authorized). Plumbing stays so it is a one-line flip; for authorized-but-not-owned sites, confirm the owner accepts this.
  • Rate limits per domain, enforced centrally (generated code cannot bypass — see §6). Polite backoff on 429 is the first access-recovery rung — good-citizen behaviour, not evasion.

2Research grounding

Two research passes (tool landscape; validation & drift literature) ran before this design. What matters:

  • The pattern is commercially proven, OSS gap is real. Kadoa, Reworkd, and Oxylabs OxyCopilot all sell exactly this loop ("AI compiles extractor once, runtime executes AI-free, AI regenerates on breakage"). No open-source project ships the full loop for data extraction — Skyvern/Stagehand do it only for browser actions.
  • Validate in layers, cheap → expensive (§10). Schema checks catch breakage-shaped errors; only semantic checks (oracle comparison, LLM judge) catch wrong-element-right-type — e.g. a price scraped from the "Recommended" sidebar that passes every type check. This is the documented killer failure mode.
  • "LLM proposes, deterministic code decides." Every serious production writeup converges here: an LLM never gets to be the sole gate that promotes a crawler. Deterministic execution + comparison decides.
  • Generate from ≥3 sample pages, not 1 — single-sample generation produces brittle nth-child-style logic (crawl4ai docs, Oxylabs uses 3–5 URLs).
  • Candidate ensembles beat single-shot — Stanford's Evaporate (VLDB'24): generate several candidate extractor functions, score, pick best — beats direct LLM extraction at ~110× fewer tokens. AutoScraper (EMNLP'24) contributes the acceptance metric design.
  • v2 Access recovery has the same "save the winner, reuse it" shape as extraction. Stagehand caches the resolved selector per action and self-heals to an LLM on miss; we apply the identical idea to the fetch side — record which access strategy got through for a domain, reuse it next time, escalate only when it stops working. Stealth fetchers (Patchright) and authorized proxies are the standard legitimate tools for JS/fingerprint challenges on your own properties.
  • Useful primitives we reuse: Playwright/Patchright (fetch + stealth), parsel (extraction lib inside generated code), Pydantic v2 (schemas), litellm (LLM access). Phase 2: Scrapling-style element fingerprints for LLM-free healing.

5Crawler contract + example generated crawler

Every generated crawler implements one small interface. The engine drives pagination (so rate limiting stays central); the crawler extracts one page and reports where "next" is. All fetching goes through ctx, which is where the access layer (§6) and rate limiting live — the crawler never opens its own connection.

crawloop/contract.py (excerpt)
class CrawlResult(BaseModel):
    items: list[dict]            # validated against the family's schema downstream
    next_url: str | None = None  # engine follows it, rate-limited, up to family max_pages

class FetchContext(Protocol):
    async def fetch(self, url: str) -> str            # HTTP GET — allowlist + rate limit + access recovery enforced inside
    async def fetch_rendered(self, url: str, wait_for: str | None = None) -> str  # Playwright/Patchright, JS pages
    def absolutize(self, base: str, href: str | None) -> str | None
    def parse_money(self, raw: str | None) -> Decimal | None   # common coercion helpers
    def clean_text(self, raw: str | None) -> str | None

class Crawler(Protocol):
    family: str                  # e.g. "books.toscrape.com/product_list"
    schema_ref: str              # e.g. "Product@1"
    async def crawl(self, url: str, ctx: FetchContext) -> CrawlResult

Example — what The Loop would actually generate for the books.toscrape.com listing family:

crawlers/books_toscrape__product_list/v3.py (generated 2026-06-12 by loop run #17)
from parsel import Selector
from crawloop.contract import Crawler, CrawlResult, FetchContext

class BooksToscrapeProductList(Crawler):
    family = "books.toscrape.com/product_list"
    schema_ref = "Product@1"

    async def crawl(self, url: str, ctx: FetchContext) -> CrawlResult:
        html = await ctx.fetch(url)
        sel = Selector(html)
        items = []
        for card in sel.css("article.product_pod"):
            items.append({
                "name": card.css("h3 a::attr(title)").get(),
                "price": ctx.parse_money(card.css(".price_color::text").get()),
                "in_stock": "In stock" in (ctx.clean_text(
                    " ".join(card.css(".availability::text").getall())) or ""),
                "url": ctx.absolutize(url, card.css("h3 a::attr(href)").get()),
            })
        next_href = sel.css("li.next a::attr(href)").get()
        return CrawlResult(items=items, next_url=ctx.absolutize(url, next_href))
Why code-as-artifact works here (your choice, and it buys real power): pagination quirks, JS-rendered pages (fetch_rendered), conditional fields, list+detail two-step crawls — all expressible without inventing a spec language. The cost — auto-deploying generated code — is paid back by §6.

6Safety model + access layer

6a · What a bad generated crawler can do — nothing the ctx doesn't permit

  1. Capability injection. Generated code receives all I/O through FetchContext. It cannot construct sockets, HTTP clients, or browsers itself. ctx.fetch* internally enforces: domain allowlist → rate limit → access strategy → logging. A crawler that "goes rogue" can only request allowed pages slower than the limit.
  2. AST static gate — runs at registration AND at every module load:
    • Import allowlist: parsel, re, json, datetime, decimal, urllib.parse, crawloop.contract. Nothing else.
    • Banned nodes: eval/exec/compile, open(), __import__, getattr-with-string-literal tricks, subprocess/os/socket/requests/httpx imports, dunder attribute access.
  3. Subprocess sandbox for candidates. During testing, candidate code runs in a subprocess with wall-clock timeout (30s/page) and memory cap, fed stored HTML — no network for the offline rounds.
  4. Promoted code runs in-process for speed — it passed the AST gate and full source + diff sits in the audit log. (You picked full-auto; this is the matching posture. Flip executor.isolation = subprocess in config for max paranoia per request.)

6b · Access layer — handling blocks & anti-bot v2

When ctx.fetch* hits a block instead of HTML, it does not stop. It walks an escalating, per-domain access-strategy ladder until something gets through, then records the winner so the next fetch starts there. This is the "fix till success" you asked for — kept legitimate by the hard allowlist (the target is always one you authorized).

#StrategyClearsLegitimate because…
1Backoff + retry (exponential, jitter)429, 5xx, transientRespecting rate limits is good-citizen behaviour, the opposite of evasion
2Realistic headers / browser-like clientnaïve UA blocksStandard for any real browser traffic
3Headless browser (Playwright)JS-rendered pages, simple JS challengesRendering the page your browser would render
4Stealth browser (Patchright)fingerprint/Cloudflare JS challengesOften clears the challenge with no "solving" at all; standard for automating your own JS-heavy property
5Authenticated session (owner-supplied login / cookies / API token)401, 403, gated contentYou are presenting your own credentials for a site you're authorized on
6Owner-issued bypass token / allowlisted proxyWAF/Cloudflare on owned sitesA token/IP you configured in your own WAF — the recommended fix for your own properties
7CAPTCHA solver — opt-in, off by default, per authorized domainexplicit captchaLast rung; requires you to set captcha_solver + assert authorization for that domain

The loop records working_strategy per domain (e.g. "stealth_browser + session"), reuses it first next time, and only re-escalates if it stops working — exactly the "save the success, reuse it" pattern from the extraction side. After k failed full ladders, the domain is escalated to a human (data may pause); it is never silently dropped.

The one honest boundary. This ladder is built to get an authorized party through to their own / permissioned content — credentials you hold, tokens you issued, challenges your own WAF raises. It is not a general anti-bot defeat engine: rungs 4–7 only ever run against allowlisted domains, and the CAPTCHA rung is opt-in and requires an explicit per-domain authorization assertion. For your own sites the cleaner fix is usually rung 6 (allowlist the crawler in your WAF) rather than solving a challenge you control. This keeps the OSS repo from being a drop-in tool for hammering sites you don't own.

7Page-family routing + version ladder

A page family = one template on one site (listing page, detail page, article page…). Registry row per family:

family: books.toscrape.com/product_list
url_patterns: ["^https?://books\\.toscrape\\.com/(catalogue/category/.+/)?(page-\\d+\\.html|index\\.html)?$"]
schema_ref: Product@1
versions:                  # ordered, newest first — the "version ladder"
  - {v: 3, status: active,    promoted: 2026-06-12, success_30d: 0.99}
  - {v: 2, status: fallback,  retired:  2026-06-12, success_30d: 0.55}   # still tried if v3 fails
  - {v: 1, status: archived}                                              # rollback only
status: healthy            # healthy | degraded | regenerating | escalated
health: {runs: 1422, validation_fail_rate_7d: 0.0, last_success: …}
"Same family, different version — what happens?" (your §7 question). Versions are not just history; at runtime they form a ladder the executor walks newest → older:
  1. Try active (v3) → validate (§10). Pass → serve, bump v3's success stat. Normal, ms.
  2. v3 fails validation (drift) → try the next fallback version (v2) on the same page. If v2 validates, serve it and record that this URL still matches the older layout.
  3. No version validates → T2 fallback serves now + enqueue The Loop.
Why this matters: real sites roll out redesigns gradually or A/B test, so both old and new layouts can be live at once. A single "active version" would fail ~half the time during a migration. The ladder absorbs that for free. If the Loop sees both layouts persist, it keeps two versions active (ordered by recent success) rather than fighting over one — and can split into sub-families if they diverge for good. A promote is therefore an append + reorder of this ladder, never a destructive swap.
  • POC Routing = URL regex match, written by The Loop when the family is first created (LLM proposes the pattern from sample URLs; you can hand-edit).
  • Phase 2 Unknown-URL handling: DOM path-shingle MinHash similarity vs known families (research: shingling beats tree-diff on cost and quality). Same primitive doubles as the structural-drift early alarm.
  • URL matches nothing → new-family flow: T2 extracts immediately (needs a target schema in the request), Loop bootstraps a brand-new crawler + registry entry. Same machinery as healing — new family is just "healing from nothing."

8Runtime flow & failure taxonomy

Online request, your "fallback OK" mode:

  1. Route url → family. Load the version ladder (§7).
  2. Run T0 ladder → validate (§10 runtime gates). Pass → return data (+ store HTML snapshot every Nth run as future fixture; reinforce the winning version's health).
  3. Fail → classify the failure (table below). Classification decides retry vs access-recovery vs regenerate. This is the policy you asked about.
  4. Drift → T2 LLM fallback serves this request now (validated against the same schema) → enqueue The Loop (deduped: one in-flight job per family). Blocked → Access Recovery (§6b) tries to get through, then re-runs the crawler.
  5. Loop / Access Recovery save their winners → subsequent requests are fast again.

Timeout, connection reset, HTTP 5xxtransientRetry ×3 w/ exponential backoff. Never regen — regen on flaky network would thrash.
HTTP 200 + schema/validation failure, empty result, fill-rate collapsedriftTry older versions in the ladder → else T2 fallback now + enqueue The Loop.
HTTP 429blocked (rate) v2Access Recovery rung 1: exponential backoff + retry; lower effective RPS for the domain.
HTTP 401 / 403, gated contentblocked (auth) v2Access Recovery: inject configured session/credentials/API token for that authorized domain; retry.

9The Loop — regeneration state machine

  1. SAMPLE. Collect ≥3 URLs of the family (recent run history; or follow links from a listing page). Fetch fresh HTML (through the access layer — so a blocked family is unblocked first). Store snapshots. One sample → brittle selectors; three+ forces generalization.
  2. ORACLE. T2 LLM direct-extracts each sample against the family schema → "oracle JSONs". These are what correct means for this run. Oracle outputs must themselves pass schema validation; if oracle can't extract, escalate (page may be gone — don't generate against garbage).
  3. CODEGEN. Strong model via litellm. Prompt (template in prompts/codegen.j2) contains: contract source, target Pydantic schema, the 3+ trimmed HTML samples (300KB → ~5KB keeping id/class/data-*/aria attrs), oracle JSONs, previous version + its failure report ("price null on 3/3 samples"). Emits 1–3 candidate modules (ensemble — Evaporate result).
  4. GAUNTLET — deterministic, LLM not consulted: AST gate → sandbox-run on stored sample HTML → per-field compare vs oracle (exact for stable fields, tolerant for volatile) → regression on golden fixtures. Score per candidate (precision/recall vs oracle, fixture pass, exec errors). Promote needs 100% schema validity, ≥98% field agreement w/ oracle, 100% fixture pass (volatile fields excluded from exactness).
  5. PROMOTE. Best passing candidate → crawlers/<family>/v<N+1>.py; the version ladder gets the new version appended and reordered to active (§7). Old versions retained. Samples + agreed outputs become new golden fixtures. Audit entry written.
  6. Retry loop. All candidates fail → regenerate with the scored failure report appended (model sees why). Max 3 rounds → family status escalated, human notified, T2 fallback keeps serving meanwhile.
Access Recovery is the same machine on the fetch side. Identical shape: try strategies in order → first that passes (a validating fetch) wins → save the winner → escalate to human after k failed rounds. One mental model, two surfaces (extraction breaks; fetching is blocked).

schemas/product.py (example — note volatile-field metadata used by gates 4/5)
class Product(BaseModel):
    model_config = ConfigDict(extra="forbid")

    name: str = Field(min_length=1)
    price: Decimal = Field(gt=0, lt=1_000_000)
    currency: str = Field(pattern=r"^[A-Z]{3}$", default="GBP")
    in_stock: bool
    url: HttpUrl
    image_url: HttpUrl | None = None

    VOLATILE: ClassVar[set[str]] = {"price", "in_stock"}   # tolerance, not exactness, in regression gates
Known residual risk (research-confirmed, accepted): if the page itself moves meaning (e.g. price now shown ex-VAT), oracle and crawler can agree on a value that's wrong for your business. Layer 5 tolerance-band alerts catch the jump; final defense is the audit log + Phase-2 monitors. No autonomous system closes this fully.

11Schema extensibility (your open-source requirement)

Adding a new output schema = drop a Pydantic file in schemas/, no engine changes:

# schemas/job_posting.py — contributed by a user
class JobPosting(BaseModel):
    model_config = ConfigDict(extra="forbid")
    title: str
    company: str
    location: str | None
    salary_min: Decimal | None
    posted_at: date | None
    url: HttpUrl
    VOLATILE: ClassVar[set[str]] = {"salary_min"}
  • Schemas are versioned by name: JobPosting@1. Families bind to a schema ref; changing a schema version triggers regen for bound families (the Loop handles migration — same machinery again).
  • The codegen + oracle prompts take the schema's JSON Schema dump — so new schemas need zero prompt changes.
  • Request API: crawl(url, schema="JobPosting@1") for new families; known families use their binding.
